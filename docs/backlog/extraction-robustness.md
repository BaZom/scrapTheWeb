# Extraction robustness — make the saved run agree with the builder

**Roadmap phase:** 0 (core loop). **Status:** open — diagnosis verified, plan proposed, **awaiting
go before code**. (When we commit to the fix this becomes an ADR + reference updates and this
ticket closes.)

## Why this is Phase 0, not polish

Skrowt's value is a *trustworthy* structured diff. Today the page the user builds against and the
page the saved run extracts are parsed and matched by **two different engines**, so they
routinely disagree: "the preview looked right but the run got different or empty data," and tables
break entirely. Selector-drift recovery (ADR 0014) now stops a broken run from lying, but it
*quarantines* these as `needs_attention` — every divergence below is a sprout that gets flagged
or returns wrong data. Fixing the engine split removes the cause, not just the symptom.

## Root cause: build engine ≠ run engine

| | Build-time (builder + snapshot preview) | Run-time (saved run, authoritative) |
|---|---|---|
| DOM source | real browser — `document.querySelectorAll("body *")` (`render_scripts/dom_candidates.js:10`) | full HTML re-parsed by a hand-rolled Python `HTMLParser` (`recipe_runner.py:22-59`) |
| Node set | **visible-only** (`dom_candidates.js:25`), **truncated to 900** (`worker.py:68`) | every node, untruncated |
| Selector match | `selector_generator.py` matcher — child `>` only (`:215`), `nth` from the browser attribute (`:427-429`) | `recipe_runner.py` matcher — descendant + child, `nth` recomputed from the Python tree (`:195-199`) |
| Field text | browser `innerText`, whitespace-collapsed (`dom_candidates.js:28`) | Python `textContent` join (`recipe_runner.py:218-225`) |

The selector is **generated by one engine and applied by another** — they don't have to agree, and
on real HTML they don't.

## Verified reproductions

Run against the real `recipe_runner` (pure module) on host Python — see the matching outputs:

1. **Implicit `<tbody>` — table listings break outright.** Browsers insert `<tbody>`; `HTMLParser`
   does not. A selector generated against the browser DOM, `table > tbody > tr`, matches **0** rows
   at run time (browser: 2). `table > tr` matches 2 in Python but 0 in the browser. Either way the
   run and the builder disagree → the sprout extracts nothing.
2. **Optional-closing `<li>` — wrong `nth-of-type`.** For `<ul><li>A<li>B<li>C</ul>` (valid HTML),
   `li:nth-of-type(2)` matches **0** in Python (browser: 1): `HTMLParser` nests the `<li>`s instead
   of treating them as siblings, so every `nth` the builder computed is off.
3. **Text value divergence.** `<div><span>HIDDEN</span>Visible<p>Line2</p></div>` →
   Python yields `"HIDDENVisibleLine2"` (hidden text included, no block breaks); the browser
   `innerText` the builder used would skip `display:none` and break blocks. Different string ⇒ the
   diff reports false "changed", and dedup keys (`worker.py _record_key`) churn.

Symptom mapping: **"sites break"** = #1/#2 (parser-tree divergence on tables and optional-close
tags); **"data inconsistent"** = #3 plus the visible-only/900-node truncation gap (build `matchCount`
is computed over a different node set than the run sees).

## Proposed fix — unify on the browser engine

Extract at **run time in the browser** via `page.evaluate` + `querySelectorAll`/`querySelector` and
`innerText`/`getAttribute` — the **same engine the builder already uses** — instead of re-parsing
HTML in Python. Preview and run then agree by construction, malformed HTML is handled exactly as a
browser handles it (because it is one), and the fragile Python parser+matcher can be retired. The
generated selectors are already valid CSS, so they run unchanged.

Render-robustness wins that help regardless of the above:
- **Auto-scroll to trigger lazy-load**, and capture HTML *after* it — today `page.content()` is taken
  before any scroll (`worker.py:457`), so below-fold/`data-src` content the user picked from the
  full-page screenshot isn't in the run's HTML.
- **Wait for the container selector to actually appear** before snapshotting (beyond the current 1s
  settle), so SPA listings aren't captured half-built.
- **`innerText` parity** for field values (drops once extraction moves into the browser).

### Suggested phasing (agree before code)
- **Phase A — render robustness** (lower risk, independently verifiable): auto-scroll + capture-after,
  wait-for-container. No engine change.
- **Phase B — engine unification**: move run extraction into the browser; migrate the legacy
  `/preview` path (`page_sessions.py:467`) and tests; **delete `recipe_runner.py`**'s parser+matcher.

## Boundaries (unchanged constraints)
- Still our render, behind the existing SSRF guards; no CAPTCHA bypass, no proxies, item-facts only
  (ADR 0012). This is a fidelity fix, not a reach-more-sites change.
- Builder guardrails unchanged: no selectors/code shown; correction is by example.

## Migration surface (for Phase B)
`recipe_runner.extract_preview_rows` is used by: the saved run (`worker.py:254`), the legacy
`/preview` endpoint (`page_sessions.py:467`), and `tests/test_recipe_runner.py`. All three migrate to
the browser-eval extractor; the snapshot preview (`selector_generator.preview_from_snapshot`) can
stay for the fast no-fetch build preview, or also converge — decide in Phase B.

## Acceptance
- A table-based listing and an optional-close/`<li>` listing that the builder maps successfully also
  extract the **same** rows on a saved run (repro #1/#2 no longer diverge).
- The same field yields the **same string** in preview and run (repro #3).
- Lazy/below-fold items the user can pick from the screenshot are present in the run.
- When shipped: record the why in an ADR, update `architecture.md`/`builder.md`, remove this ticket.
